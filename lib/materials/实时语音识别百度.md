百度实时语音识别 - websocket API 技术文档
更新时间：2025-11-26
接口描述
实时语音识别接口采用 websocket 协议的连接方式，边上传音频边获取识别结果。可以将音频流实时识别为文字，也可以上传音频文件进行识别；返回结果包含每句话的开始和结束时间，适用于长句语音输入、音视频字幕、直播质检、会议记录等场景。
WebSocket 简介
WebSocket 是基于 TCP 的全双工协议，即建立连接后通讯双方都可以不断发送数据。WebSocket 协议由 rfc6455 定义，一般需要各编程语言的 WebSocket 库来实现接入，且 WebSocket 库需支持 rfc6455 描述的协议，即支持Sec-WebSocket-Version: 13。
主要流程
客户端与服务端交互流程
开始阶段：建立 WebSocket 连接 → 确认建立 → 发送开始报文
识别阶段：发送实时音频报文 → 接受实时识别结果报文
结束阶段：发送结束报文 → 关闭 WebSocket 连接 → 关闭成功

连接	这里指 TCP 连接及握手（Opening Handshake），一般 WebSocket 库已经封装，用户不必关心
发送数据帧	Sending Data Frame，类似包的概念，指一次发送的内容，从客户端到服务端

接收数据帧	Receiving Data Frame，类似包的概念，指一次发送的内容，从服务端到客户端

- 二进制帧：实时语音识别 api 不会收到
关闭连接	Closing Handshake。百度服务端识别结束后会自行关闭连接，部分 WebSocket 库需要得知这个事件后手动关闭客户端连接

发送音频数据帧
核心注意点：服务端 5s 没有收到音频数据会断开并报错，发送间隔需控制在 5s 以内
帧类型：Opcode 为 Binary
内容：二进制的音频内容
帧长度要求：除最后一个音频数据帧，每个帧的音频数据长度为 20-200ms；建议最佳 160ms 一个帧，有限制的也建议 80ms
160ms 音频数据字节数计算：
plaintext
计算方式：
16000采样率： 1s音频 16000采样点
16bits： 一个采样点 16bits = 2 bytes
1s ： = 1000ms
即 160ms *  16000  * 2bytes / 1000ms = 5120bytes
发送规则：
实时发送音频：每个 160ms 的帧之后，下一个音频数据帧需要间隔 160ms（如读取文件发送，需sleep(160ms)）
网络异常补发：发送的音频数据帧之间可以没有间隔，具体见「断网补发数据」章节

返回结果类型及示例
1. 临时识别结果（MID_TEXT）
一段音频由多句话组成，实时识别 api 会依次返回每句话的临时识别结果，示例：
json
{
    "err_no":0,
    "err_msg":"OK",
    "type": "MID_TEXT",
    "result": "北京天气怎",
    "log_id": 45677785,
    "sn": "399427ce-e999-11e9-94c8-fa163e4e6064_ws_2"
}
2. 最终识别结果（FIN_TEXT）
每句话的最终识别结果包含开始和结束时间，示例：
json
{
    "err_no":0,
    "err_msg":"OK",
    "log_id":45677785,
    "sn":"399427ce-e999-11e9-94c8-fa163e4e6064_ws_2",
    "type":"FIN_TEXT",
    "result":"北京天气怎么样",
    "start_time":53220,
    "end_time":73340,
    "product_id":1537,
    "product_line":"open"
}
3. 心跳帧（HEARTBEAT）
服务端会 5s 下发一次心跳帧，收到后忽略即可，示例：
json
{
    "type":"HEARTBEAT"
}
4. 服务端报错
错误码针对单个句子，其它句子依旧可以识别，请求是否结束以服务端是否关闭连接为准，示例：
json
{
    "err_msg": "asr authentication failed",
    "err_no": -3004,
    "log_id": 106549729,
    "result": "",
    "sn": "0d25b5e1-ffa6-11e9-8968-f496349f3917",
    "type": "FIN_TEXT"
}
一句话的开始与结束时间说明
识别结果中，仅FIN_TEXT类型的返回结果包含start_time（开始时间）和end_time（结束时间），单位为毫秒
音频时间结构：[静音][s1:e1][静音][s2:e2]....[静音][s10:e10][静音]（s1 e1 指第一句话的开始和结束时间）
长度限制：一句话不能超过 60s，否则会被强制切割

断网补发数据
补发目的
将一个语音流在网络不佳的情况下，通过代码逻辑拼接，使得多次请求的结果看上去像一次连续的识别结果。
核心原则
哪里断开，从哪里开始重新发一次请求，断开位置 = 最后一次接收的end_time；服务端对每个请求独立，需要自行拼接补发数据的请求时间。
场景示例
若一个音频的 10 个句子时间为[静音][s1:e1][静音][s2:e2]...[s7:e7]...[静音][s10:e10][静音]，在第七句 e7 之后网络抖动，需从 e7 开始发起新请求，音频数据帧之间无需 sleep，保证连续补发。
补发注意点
若超过 5s 没有发送音频数据给服务端，服务端会下发报错并结束连接，建议至少 2s 发送一次数据。
完整补发流程
首次请求：正常发送音频数据，并缓存音频数据
正常接收：接收到数据帧后，保存end_time
网络断开：读取最后一次保存的end_time（如 7000ms）
发起新请求：从缓存的音频数据中找到end_time对应的位置（7000ms 对应 224000bytes）以后的数据，发送给服务端；每个帧按 160ms 音频数据处理，补数据时帧之间无需间隔 sleep
追更实时：一直发送至追上实时音频数据后，恢复正常的实时发送规则（帧间隔 160ms）
时间拼接：新请求返回的start_time和end_time需加上之前的end_time（如 7000ms），再展示给用户
补发收尾：若补发数据过大导致新请求过快结束，在新请求结束时需补发type=HEARTBEAT心跳帧，建议 2-3s 发一次，避免 5s 服务端读超时
异常重入：如果再次断网，重复上述流程
