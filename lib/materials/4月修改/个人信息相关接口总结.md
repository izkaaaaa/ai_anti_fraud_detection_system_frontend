# 个人信息相关接口总结

> 适用范围：用户个人资料、用户画像、监护人信息、家庭绑定信息相关接口。
> 
> 说明：本文重点总结“个人信息”相关接口，不展开实时检测、通话记录等业务接口。

---

## 1. 接口范围说明

结合项目代码，和“个人信息”最相关的接口主要集中在 `app/api/users.py`，包括：

1. 获取当前用户信息
2. 更新用户画像/个人资料
3. 获取监护人信息
4. 解绑家庭组
5. 注册时填写的个人基础信息
6. 登录后获取当前用户资料
7. 个人安全报告（基于用户本人历史记录生成）

---

## 2. 个人信息核心接口总览

| 分类 | 接口 | 方法 | 用途 |
|------|------|------|------|
| 当前用户信息 | `/api/users/me` | `GET` | 获取当前登录用户的个人信息 |
| 更新个人画像 | `/api/users/profile` | `PUT` | 更新角色、性别、职业、婚姻状况 |
| 监护人信息 | `/api/users/guardian` | `GET` | 获取当前用户的监护人/家庭管理员信息 |
| 家庭解绑 | `/api/users/family` | `DELETE` | 解除当前用户与家庭组的绑定 |
| 用户注册 | `/api/users/register` | `POST` | 创建用户时录入基础个人信息 |
| 用户登录 | `/api/users/login` | `POST` | 登录并返回用户资料 |
| 发送注册验证码 | `/api/users/send-code` | `POST` | 注册前发送邮箱验证码 |
| 发送登录验证码 | `/api/users/send-login-code` | `POST` | 邮箱验证码登录前发送验证码 |
| 个人安全报告 | `/api/users/{user_id}/security-report` | `GET` | 生成用户个人安全监测报告 |

---

## 3. 用户数据结构说明

从用户模型与 Schema 来看，项目中的个人信息主要包含以下字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `user_id` | int | 用户 ID |
| `phone` | string | 手机号 |
| `email` | string | 邮箱 |
| `username` | string | 用户名 |
| `name` | string | 真实姓名/昵称 |
| `family_id` | int | 所属家庭组 ID |
| `role_type` | string | 角色类型，如老人、儿童、学生、青壮年 |
| `gender` | string | 性别 |
| `profession` | string | 职业 |
| `marital_status` | string | 婚姻状况 |
| `is_active` | bool | 账号是否启用 |
| `is_admin` | bool | 是否家庭管理员 |
| `created_at` | datetime | 创建时间 |

其中：
- `phone`、`username` 是用户基础身份字段
- `role_type`、`gender`、`profession`、`marital_status` 属于用户画像字段
- `family_id`、`is_admin` 与家庭关系和监护人逻辑有关

## 4. 个人信息查询接口

## 4.1 获取当前用户信息

**接口**：`GET /api/users/me`

### 作用

获取当前登录用户自己的资料。

这个接口是个人中心、资料页最核心的读取接口。

### 认证方式

需要登录后携带 Token。

### 返回内容

返回类型是 `UserResponse`，包含当前用户的主要个人信息。

### 典型返回示例

```json
{
  "user_id": 1,
  "phone": "13800138000",
  "username": "zhangsan",
  "name": "张三",
  "role_type": "青壮年",
  "gender": "男",
  "profession": "教师",
  "marital_status": "已婚",
  "family_id": 3,
  "is_active": true,
  "created_at": "2026-04-04T10:00:00"
}
```

### 适用场景

- 个人中心初始化
- 编辑资料前回显
- 判断当前用户是否已绑定家庭组
- 获取当前用户画像用于推荐、检测、告警个性化展示

---

## 5. 个人信息更新接口

## 5.1 更新用户画像

**接口**：`PUT /api/users/profile`

### 作用

更新当前用户的画像信息。

注意：这个接口目前更新的是**画像字段**，不是所有基础资料都能修改。

### 可更新字段

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `role_type` | string | 否 | 角色类型 |
| `gender` | string | 否 | 性别 |
| `profession` | string | 否 | 职业 |
| `marital_status` | string | 否 | 婚姻状况 |

### 请求示例

```json
{
  "role_type": "老人",
  "gender": "女",
  "profession": "退休",
  "marital_status": "已婚"
}
```

### 返回示例

```json
{
  "code": 200,
  "message": "用户画像更新成功",
  "data": {
    "user_id": 1,
    "role_type": "老人"
  }
}
```

### 特点说明

1. 接口采用“按字段局部更新”方式
2. 哪个字段传了就更新哪个字段
3. 没传的字段保持原值不变

### 当前限制

该接口**不会**更新以下字段：
- 手机号
- 邮箱
- 用户名
- 姓名
- 密码

如果后续你需要，我可以再帮你补一份“当前项目里哪些个人字段能改、哪些不能改”的说明文档。

## 6. 家庭/监护人信息接口

## 6.1 获取当前用户的监护人信息

**接口**：`GET /api/users/guardian`

### 作用

获取当前用户所在家庭组中的监护人信息。

后端逻辑会：
1. 先根据当前用户查出 `family_id`
2. 再去家庭管理员表里查同家庭组的管理员
3. 排除当前用户自己
4. 返回监护人列表

### 认证方式

需要登录。

### 返回情况 1：用户未绑定家庭组

```json
{
  "code": 200,
  "message": "未绑定家庭组",
  "data": {
    "guardian": null
  }
}
```

### 返回情况 2：家庭组内未设置监护人

```json
{
  "code": 200,
  "message": "家庭组内未设置监护人",
  "data": {
    "guardian": null
  }
}
```

### 返回情况 3：成功获取监护人

```json
{
  "code": 200,
  "message": "获取监护人成功",
  "data": {
    "guardians": [
      {
        "user_id": 2,
        "name": "李四",
        "phone": "13900139000",
        "admin_role": "primary"
      }
    ]
  }
}
```

### 字段说明

| 字段 | 说明 |
|------|------|
| `user_id` | 监护人用户 ID |
| `name` | 监护人姓名，没有则退回用户名 |
| `phone` | 监护人手机号 |
| `admin_role` | 在家庭组中的管理员角色 |

---

## 6.2 解绑家庭组

**接口**：`DELETE /api/users/family`

### 作用

解除当前用户与家庭组的绑定，本质上是把用户的 `family_id` 设为 `null`。

### 返回示例

```json
{
  "code": 200,
  "message": "解绑成功",
  "data": {
    "user_id": 1
  }
}
```

### 适用场景

- 用户主动退出家庭组
- 重置绑定关系
- 个人信息/成员管理中的解绑操作

---

---

## 7.2 用户注册

**接口**：`POST /api/users/register`

### 作用

创建新用户，同时录入初始个人资料。

### 注册时可填写的个人信息字段

| 字段 | 必填 | 说明 |
|------|------|------|
| `phone` | 是 | 手机号 |
| `email` | 是 | 邮箱 |
| `username` | 是 | 用户名 |
| `name` | 否 | 姓名 |
| `password` | 是 | 密码 |
| `email_code` | 是 | 邮箱验证码 |
| `role_type` | 否 | 角色类型 |
| `gender` | 否 | 性别 |
| `profession` | 否 | 职业 |
| `marital_status` | 否 | 婚姻状况 |

### 请求示例

```json
{
  "phone": "13800138000",
  "email": "user@example.com",
  "username": "zhangsan",
  "name": "张三",
  "password": "123456",
  "email_code": "1234",
  "role_type": "青壮年",
  "gender": "男",
  "profession": "教师",
  "marital_status": "已婚"
}
```

### 返回示例

```json
{
  "code": 201,
  "message": "注册成功",
  "data": {
    "user_id": 1
  }
}
```

### 业务校验

该接口会校验：
- 邮箱验证码是否正确
- 手机号是否已注册
- 邮箱是否已注册
- 用户名是否已被使用

## 8. 登录后返回的个人信息接口

## 8.1 用户登录

**接口**：`POST /api/users/login`

### 支持的登录方式

1. 邮箱 + 验证码
2. 邮箱 + 密码
3. 手机号 + 密码

### 登录请求示例 1：邮箱 + 密码

```json
{
  "email": "user@example.com",
  "password": "123456"
}
```

### 登录请求示例 2：邮箱 + 验证码

```json
{
  "email": "user@example.com",
  "email_code": "1234"
}
```

### 登录请求示例 3：手机号 + 密码

```json
{
  "phone": "13800138000",
  "password": "123456"
}
```

### 返回示例

```json
{
  "access_token": "xxxxx",
  "token_type": "bearer",
  "user": {
    "user_id": 1,
    "phone": "13800138000",
    "username": "zhangsan",
    "name": "张三",
    "role_type": "青壮年",
    "gender": "男",
    "profession": "教师",
    "marital_status": "已婚",
    "family_id": 3,
    "is_active": true,
    "created_at": "2026-04-04T10:00:00"
  }
}
```

### 说明

登录接口除了返回 Token，还会直接返回用户信息，因此也可以作为 App 登录后初始化个人资料的来源之一。

## 9. 与个人信息相关的报告接口

## 9.1 生成用户专属安全报告

**接口**：`GET /api/users/{user_id}/security-report`

### 作用

分析该用户近期通话记录，生成个性化安全报告。

虽然它不是“编辑个人信息”的接口，但它是明显围绕“某个用户本人”的个人化接口，因此也可归入个人信息相关能力。

### Query 参数

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `stream` | boolean | 否 | 是否流式输出，默认 `false` |

### 返回方式

#### 1）非流式返回

直接返回完整报告 JSON，包括：
- `user_id`
- `username`
- `report_generated_at`
- `report_content`
- `stats`

#### 2）流式返回

使用 SSE 分段返回报告内容，适合前端边生成边展示。

### 典型用途

- 个人安全月报
- 用户风险画像展示
- 家庭管理员查看成员安全情况

## 10. 推荐给前端重点接入的个人信息接口

如果是做“个人中心 / 我的资料 / 家庭绑定”页面，最值得优先接入的是下面几个：

### 必接接口

1. `GET /api/users/me`
2. `PUT /api/users/profile`
3. `GET /api/users/guardian`
4. `DELETE /api/users/family`

### 注册登录阶段

5. `POST /api/users/send-code`
6. `POST /api/users/register`
7. `POST /api/users/send-login-code`
8. `POST /api/users/login`

### 个性化扩展接口

9. `GET /api/users/{user_id}/security-report`

## 11. 结论

如果只从“个人信息”角度看，这个项目最核心的是三类能力：

### 1）读取个人资料
- `GET /api/users/me`

### 2）更新个人画像
- `PUT /api/users/profile`

### 3）查询家庭/监护人关系
- `GET /api/users/guardian`
- `DELETE /api/users/family`

而注册、登录接口则负责个人基础信息的首次录入和登录态建立。
