# 荣耀设备通话录音完整机制分析（终版）

> 基于对通话录音Pro源码的完整逆向分析，记录所有关键发现，供AI诈骗识别项目参考实现。

---

## 核心结论

通话录音Pro在荣耀设备上能录到QQ通话声音，依赖以下四个组件的协同配合：

| 组件 | 作用 |
|------|------|
| `VOICE_COMMUNICATION` 声音源 | 直接读取系统通话音频流，不是MIC |
| 无障碍服务 | 检测通话开始，执行系统级操作 |
| 1像素Activity | 欺骗荣耀系统认为应用有前台界面，保住音频访问权限 |
| 悬浮窗 | 录音期间唯一的用户交互入口 |

**缺少任何一个，在荣耀设备上都会失败。**

---

## 完整事件时间线

### T0：应用初始化

- 用户首次打开通话录音Pro
- 弹出引导，要求用户手动开启无障碍服务
- 用户在系统设置中开启 → `RecordHelperService.onServiceConnected()` 触发
- `KeepAliveService` 启动，创建**前台通知**，保证进程不被系统杀死
- 应用就绪，等待通话

---

### T1：用户点击「开始检测」

- 悬浮窗通过 `WindowManager` 添加到屏幕上层（系统级浮层，不属于任何Activity）
- 通话录音Pro主界面退到后台
- 录音器尚未启动，处于监听等待状态

---

### T2：用户打开QQ，发起或接听视频通话

- QQ通话界面进入前台
- **无障碍服务触发** `onAccessibilityEvent`（`eventType = TYPE_WINDOW_STATE_CHANGED = 32`）
- 读取事件中的包名和类名，匹配到 QQ 通话相关 Activity
- 调用 `OooOO0O.OooO0Oo().OooO0o0()` → **悬浮窗弹出录音控制界面，录音正式开始**
- 录音器以 **`AudioSource.VOICE_COMMUNICATION`（值=6）** 初始化 `AudioRecord`
- 采样参数：16000Hz，16bit，单声道
- `f7645OooOOOo = true`，标记为正在录音

---

### T3：录音进行中——荣耀专属闭环机制启动

#### 场景A：用户在QQ通话、小窗、其他App之间切换

- `onAccessibilityEvent` 触发
- 检测到荣耀设备 + `f7645OooOOOo == true`
- 读取通话录音Pro自己的 Activity 栈：`app.f7159OooO0O0.getLast()`
- **此时栈顶不是通话录音Pro的Activity（用户在QQ或其他界面）**
- → **什么都不做，用户可以自由切换**
- QQ可以小窗，可以切到其他App，完全不受影响

#### 场景B：用户试图切回通话录音Pro主界面

- `onAccessibilityEvent` 触发
- 检测到荣耀设备 + `f7645OooOOOo == true`
- 读取Activity栈：`last = MainActivity`（通话录音Pro自己的界面出现了）
- → **立刻从 `last`（MainActivity）启动 `OnePXActivity`**
- `OnePXActivity.onCreate()` 执行：
  - 检测到 `f7645OooOOOo == true`（正在录音）
  - 把自己缩成 **1像素**，放到屏幕左上角（用户看不到）
  - **500ms 后执行 `performGlobalAction(GLOBAL_ACTION_BACK)`，模拟系统返回键**
  - 用户被强制打回上一个界面
- 这个返回动作再次触发窗口切换事件 → 回到场景A或场景B，形成闭环

**结论：用户可以在任意其他界面自由切换，唯独切不回通话录音Pro主界面。**

---

### T4：为什么 VOICE_COMMUNICATION 能被成功读取

荣耀系统在通话时会对后台应用的音频权限做额外限制，但此时的应用状态是：

```
前台通知（KeepAliveService）          → 系统认为应用有前台服务
1像素Activity（OnePXActivity）        → 系统认为应用有前台界面
无障碍服务（RecordHelperService）     → 系统级权限，不受后台限制
```

三重条件同时满足，荣耀系统判定该应用为「活跃前台应用」，**不封锁其音频访问权限**，录音服务得以持续读取 `VOICE_COMMUNICATION` 通话音频流。

---

### T5：用户点击悬浮窗「停止录音」

- `f7645OooOOOo = false`，标记为停止录音
- `onAccessibilityEvent` 再次触发时，检测到不再录音
- `last instanceof OnePXActivity` → 调用 `last.finish()`，销毁1像素Activity
- 录音文件保存完成（WAV格式）
- 悬浮窗恢复待机状态
- 用户可以正常切回通话录音Pro主界面查看录音

---

## 两个机制的职责区分（重要）

很多人会混淆这两个机制，这里明确区分：

### 1像素Activity 的职责
**保住音频权限**
- 让荣耀系统认为应用有前台界面
- 防止系统在后台封锁 `VOICE_COMMUNICATION` 访问权限
- 与「用户能不能切换界面」**无关**

### performGlobalAction(BACK) 闭环 的职责
**防止主界面抢占前台**
- 只在用户试图切回通话录音Pro**主界面**时触发
- 强制把主界面打回去，防止主界面Activity覆盖录音所需的状态
- 用户切到QQ小窗、其他App，**完全不触发**

---

## 声音源的关键区别

```
AudioSource.MIC（值=1）
  → 录制物理麦克风环境声
  → 荣耀通话时后台会被完全封锁
  → 我们项目最初用的是这个，所以录不到

AudioSource.VOICE_COMMUNICATION（值=6）
  → 专为VoIP通话设计的系统音频流
  → 可以直接读取到通话双方的声音
  → 配合三重保活机制，荣耀系统允许访问
  → 通话录音Pro用的是这个
```

## 对我们项目的实现清单

### Android原生端必须实现：

- [ ] 录音器使用 `AudioSource.VOICE_COMMUNICATION`，不要用 `MIC`
- [ ] `RecordHelperService`：继承 `AccessibilityService`，监听 `TYPE_WINDOW_STATE_CHANGED` 事件，识别QQ/微信通话Activity，触发录音
- [ ] `OnePxActivity`：检测到录音中时缩成1像素，500ms后执行 `performGlobalAction(GLOBAL_ACTION_BACK)`
- [ ] `KeepAliveService`：前台服务 + 通知，保活
- [ ] 悬浮窗：录音状态显示 + 停止按钮
- [ ] 荣耀/华为设备判断：`Build.BRAND.lowercase() == "honor" || == "huawei"`
- [ ] 录音参数：16000Hz，16bit，单声道

### 触发逻辑：

```kotlin
override fun onAccessibilityEvent(event: AccessibilityEvent?) {
    val isRecording = RecordingManager.isRecording
    val brand = Build.BRAND.lowercase()
    val isHuawei = brand == "huawei" || brand == "honor"

    // 荣耀/华为：防止主界面切回
    if (isHuawei && isRecording) {
        val lastActivity = App.activityStack.lastOrNull()
        if (lastActivity != null && lastActivity !is OnePxActivity) {
            // 通话录音Pro自己的界面出现了，立刻启动1像素Activity
            val intent = Intent(this, OnePxActivity::class.java)
            lastActivity.startActivity(intent)
        }
    }

    // 通用：检测QQ通话开始
    if (event?.eventType == AccessibilityEvent.TYPE_WINDOW_STATE_CHANGED) {
        val className = event.className?.toString() ?: return
        val packageName = event.packageName?.toString() ?: return

        if (isQQVideoCall(packageName, className) ||
            isWeChatVideoCall(packageName, className)) {
            RecordingManager.startRecording()
        }
    }
}
```

### OnePxActivity核心逻辑：

```kotlin
override fun onCreate(savedInstanceState: Bundle?) {
    super.onCreate(savedInstanceState)

    if (!RecordingManager.isRecording) {
        // 没在录音，正常跳主界面
        startActivity(Intent(this, MainActivity::class.java))
        finish()
        return
    }

    // 缩成1像素
    val params = window.attributes
    params.width = 1
    params.height = 1
    params.gravity = Gravity.LEFT or Gravity.TOP
    params.x = 0
    params.y = 0
    window.attributes = params

    // 500ms后执行返回键
    Handler(Looper.getMainLooper()).postDelayed({
        // 通过无障碍服务执行
        AccessibilityServiceManager.performBack()
    }, 500)
}
```

## 总结

通话录音Pro钻的空子可以用一句话概括：

**用1像素Activity骗过荣耀系统的前台检测，用 `VOICE_COMMUNICATION` 直读通话音频流，用无障碍服务的返回键闭环防止主界面破坏录音状态，用悬浮窗作为唯一交互入口——四个组件缺一不可，共同构成了一套完整的通话录音方案。**
